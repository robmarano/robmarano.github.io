\begin{frame}{1. MIPS ISA Review}
\begin{enumerate}
    \item \textbf{Simplicity favors regularity}: Fixed instruction sizes (32 bits).
    \item \textbf{Smaller is faster}: Limited register file (32 general purpose registers).
    \item \textbf{Make the common case fast}: Optimize arithmetic and data movement.
    \item \textbf{Good design demands good compromises}: E.g., introducing the limited I-Type format to handle immediate constants efficiently.
\end{enumerate}
\vspace{0.2cm}
\begin{block}{Load/Store Architecture}
MIPS inherently prohibits direct memory-to-memory operations. Data MUST be \textbf{loaded} into a register, manipulated, and then \textbf{stored} back to RAM.
\end{block}
\end{frame}

\begin{frame}{2. The Supervisor \& Emulation}
A physical CPU typically runs an Operating System to manage programs. For MIPS, we use simulators like \textbf{SPIM} or \textbf{MARS}.
\vspace{0.2cm}
\begin{itemize}
    \item \textbf{Loading}: Placing instructions into the Text Segment (\texttt{0x00400000}) and statics into the Data Segment (\texttt{0x10000000}).
    \item \textbf{Execution}: Managing the Program Counter (\texttt{PC}).
    \item \textbf{System Calls (\texttt{syscall})}: Handing execution back to the emulator to perform restricted OS-level functions:
    \begin{itemize}
        \item Consuming input or printing to the terminal.
        \item Halting execution cleanly (Exit).
    \end{itemize}
\end{itemize}
\end{frame}

\begin{frame}[fragile]{3. Case Study: Quake's Fast Inverse Square Root}
The legendary algorithm used in \textit{Quake III Arena} (1999) to compute $\frac{1}{\sqrt{x}}$ for 3D lighting without expensive division circuitry.
\vspace{0.1cm}
\begin{verbatim}
# i  = 0x5f3759df - ( i >> 1 );
mfc1  $t0, $f12          # Move float to integer register (Bit Hack)
srl   $t1, $t0, 1        # i >> 1 (Logical shift right)
lw    $t2, magic         # Load magic number (0x5f3759df)
sub   $t0, $t2, $t1      # Subtract to get initial guess
mtc1  $t0, $f0           # Move bits back to float register
\end{verbatim}
\vspace{0.1cm}
\begin{itemize}
    \item \textbf{The Magic}: Exploits the IEEE 754 log-based exponent representation to rapidly approximate $x^{-0.5}$.
\end{itemize}
\end{frame}

\begin{frame}{4. Introduction to SystemVerilog (SV)}
SystemVerilog is a Hardware Description Language (HDL) used to design structured, parallel digital circuits.
\vspace{0.3cm}
\begin{itemize}
    \item \textbf{Compilation}: We use \texttt{iverilog} to aggressively synthesize models.
    \item \textbf{Simulation}: We use \texttt{vvp} as the runtime engine.
    \item \textbf{Waveforms}: Visualization generated by \texttt{GTKWave}.
\end{itemize}
\begin{block}{Data Types vs Software}
A \texttt{logic} variable models a physical wire. It isn't just an integer; it enforces a strict rule against multiple short-circuiting electrical drivers.
\end{block}
\end{frame}

\begin{frame}{5. Logic Modeling Paradigms in SV}
\begin{itemize}
    \item \textbf{Structural}: Instantiating physical logic primitives natively (e.g., \texttt{and}, \texttt{or}, \texttt{not}) and cabling them explicitly. Best for explicit delay modeling.
    \item \textbf{Dataflow}: Highly preferred continuous assignment (\texttt{assign}) using Boolean operators (\texttt{\&}, \texttt{|}, \texttt{\^}). The compiler optimizes the final gate netlist.
    \vspace{0.3cm}
    \item \textbf{Combinational Logic}: Output intimately depends \textit{only} on present inputs. Designed via \texttt{assign} or \texttt{always\_comb}. 
    \item \textbf{Sequential Logic}: Employs memory buffers reliant on historical state and a synchronous \textit{Clock}. Designed via \texttt{always\_ff}.
\end{itemize}
\end{frame}
